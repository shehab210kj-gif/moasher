# Taqreer AI / Muasher Real Estate Intelligence

## Current Real Integration Notes

This project is a React + Vite frontend with a local FastAPI prediction service for the Muasher valuation model.

### Real Dataset

The frontend currently reads the cleaned Jeddah transaction dataset directly from:

```text
src/ai_model/muasher_full_data.csv
```

The raw/Excel source used by the Python preparation pipeline is:

```text
src/ai_model/Jeddah_RealEstate_Final_Cleaned.xlsx
```

The frontend data layer in `src/lib/data.ts` uses the CSV to calculate district indicators, comparable transactions, average/median/min/max price per sqm, transaction count, and liquidity score.

### Model Artifacts

The prediction API uses the real trained model artifacts in:

```text
src/ai_model/valuation_model.xgb.json
src/ai_model/valuation_model.pkl
src/ai_model/le_neighborhood.pkl
src/ai_model/le_type.pkl
src/ai_model/le_plan.pkl
src/ai_model/plan_defaults.json
src/ai_model/prediction_stats.json
```

`/predict` prefers the native XGBoost artifact `valuation_model.xgb.json` and falls back to `valuation_model.pkl` if the native artifact cannot be loaded.

### Environment Variables

Create or update `.env` for the frontend:

```env
VITE_MODEL_API_URL=http://127.0.0.1:8000
VITE_MAPBOX_TOKEN=your_mapbox_token
VITE_SUPABASE_URL=your_supabase_url
VITE_SUPABASE_ANON_KEY=your_supabase_anon_key
```

`VITE_MAPBOX_TOKEN` is required for the Mapbox location picker and report map preview. Reports are stored in Supabase when authentication and migrations are configured, with `localStorage` as a development fallback.

### Test Full Flow

1. Start the FastAPI prediction API.
2. Start the frontend.
3. Log in through Supabase at `/login`.
4. Open `/app/new-analysis`.
5. Select a district, then choose an exact property location on the map or enter latitude/longitude manually.
6. Enter district, property type, area, asking price, and assumptions.
7. Generate the report.
8. Open the generated report from `/app/reports/:id`.

If the model API is unavailable, the frontend falls back to real CSV market statistics and marks the valuation source as `statistical_fallback`. If Supabase is not configured during development, report storage falls back to browser `localStorage`.

### Current Limitations

- The model currently uses district, property type, area, inferred plan, year, and month. Latitude/longitude, street width, land use, and corner status are accepted by the API but not yet model features.
- Comparable transactions do not include distances because the transaction CSV does not include coordinates.
- Market Explorer map uses district-level geography until transaction-level coordinates are available.
- PDF export currently uses browser print; a dedicated PDF rendering service has not been added.

منصة ويب لتحليل عقارات جدة: خريطة تفاعلية، تقييم تقديري للأرض، اشتراطات بناء، دراسة جدوى مالية، ولوحة سوق وتقرير استثماري قابل للطباعة.

هذا الملف مقصود يكون "ملخص سياقي" لأي ذكاء اصطناعي أو مطور يدخل المشروع، حتى لا يضطر يقرأ كل الملفات في كل مرة.

### Endpoints

#### `GET /`

يرجع رسالة ترحيبية ونسخة API.

#### `GET /neighborhoods`

يرجع أسماء الأحياء من `JEDDAH_REGULATIONS` في `regulations_engine.py`.

Response:

```json
{
  "neighborhoods": ["..."]
}
```

#### `GET /property-types`

يرجع قائمة أنواع عقارات ثابتة.

Response:

```json
{
  "property_types": ["قطعة أرض", "شقة", "فيلا", "عمارة"]
}
```

#### `POST /quick-analysis`

يستخدم لتحليل سريع بعد اختيار موقع.

Request body:

```json
{
  "neighborhood": "الشاطئ",
  "property_type": "قطعة أرض",
  "area": 600,
  "usage": "residential",
  "finish_level": "تشطيب_متوسط",
  "lat": 21.5433,
  "lng": 39.1925
}
```

Response fields:

```json
{
  "neighborhood": "...",
  "estimated_land_value": 1500000,
  "max_floors": 4,
  "best_use": "سكني",
  "expected_roi": 8.5
}
```

#### `POST /full-report`

يرجع التقرير الكامل المستخدم في `LandingPage` و `ReportPage`.

أهم أجزاء response:

```json
{
  "neighborhood": "...",
  "area": 600,
  "valuation": {},
  "regulations": {},
  "building_potential": {},
  "feasibility": {},
  "comparables": [],
  "location": { "lat": 21.5, "lng": 39.1 },
  "ranking": {},
  "ai_analysis": {}
}
```

#### `GET /market-stats`

مستخدم في `DashboardPage.tsx`.

ملاحظة مهمة: في النسخة الحالية من `muasher_api.py` التي تم فحصها لا يظهر تعريف endpoint باسم `/market-stats`، رغم أن الواجهة تطلبه. لو لوحة التحكم لا تعمل، هذه أول نقطة تفحصها.

## نقاط مهمة قبل أي تعديل

- لا تغير أسماء مفاتيح response من `/full-report` بدون تعديل `LandingPage.tsx` و `ReportPage.tsx`.
- `ReportPage` يعتمد على بنية nested محددة جدًا:
  - `report.valuation`
  - `report.regulations`
  - `report.building_potential`
  - `report.feasibility`
  - `report.ai_analysis`
- `DashboardPage` يحتاج `/market-stats`. لو أضفت endpoint، رجّع على الأقل:

```json
{
  "avg_price_per_meter": 3500,
  "total_transactions": 10000,
  "total_neighborhoods": 50,
  "annual_growth_rate": 6.6,
  "property_type_distribution": {
    "قطعة أرض": 5000,
    "شقة": 3000
  }
}
```

- النصوص العربية في بعض الملفات قد تظهر بشكل غير صحيح في terminal حسب الترميز. استخدم UTF-8 عند القراءة والكتابة.
- بعض النصوص داخل الكود ظاهرة حاليًا كـ mojibake؛ لو الهدف تحسين الواجهة العربية، عالج الترميز بحذر ولا تعمل replace عشوائي.
- الخرائط تستخدم `VITE_MAPBOX_TOKEN`. لا تضف tokens جديدة داخل الكود.
- المشروع فيه ملفات data كثيرة في الجذر يبدو أنها مراحل معالجة وتجارب. لا تحذفها بدون طلب صريح.

## مشاكل/ديون تقنية معروفة

- يوجد `requirements.txt` للـ Backend، ولا يوجد `pyproject.toml`.
- Endpoint `/market-stats` مطلوب من الواجهة لكنه غير موجود في `muasher_api.py` حسب الفحص الحالي.
- `valuation_model.pkl` مستخدم في `/predict`، بينما `/full-report` الأقدم ما زال rule-based.
- يستخدم مسار الموديل `VITE_MODEL_API_URL` مع fallback محلي للتطوير.
- `MapComponent.tsx` أصبح يقرأ `VITE_MAPBOX_TOKEN`، لكنه legacy وغير مستخدم كخريطة التحليل الرئيسية.
- بعض components مثل `Sidebar`, `DashboardLayout`, `BottomNav`, `KPICard` موجودة لكن ليست كلها مستخدمة في المسارات الحالية.
- لا توجد اختبارات آلية.
- `package.json` لا يزال باسم template: `magic-patterns-vite-template`.
- ملف `Project_Documentation.md` موجود لكن نصه يظهر بترميز غير صحيح في الطرفية.

## طريقة التفكير عند تعديل المشروع

لو أنت AI يعمل على هذا المشروع:

- ابدأ من `README.md` ثم اقرأ فقط الملفات المرتبطة بالطلب.
- لو الطلب Frontend، غالبًا ستحتاج `src/pages/*`, `src/components/*`, و `src/index.css`.
- لو الطلب Backend/API، اقرأ `src/ai_model/muasher_api.py` والمحركات الثلاثة.
- لو الطلب بيانات أو تدريب، اقرأ `train_muasher_model.py`, `prepare_full_data.py`, وملفات JSON الصغيرة أولًا.
- لا تفتح الملفات الكبيرة إلا لو مطلوب.
- تحقق بعد أي تعديل Frontend بـ:

```bash
npm run build
```

- تحقق بعد أي تعديل Backend بتشغيل FastAPI وتجربة endpoints.

## النشر

يوجد `vercel.json` و Vite مناسب للنشر على Vercel كواجهة أمامية. لكن الـ Backend المحلي FastAPI غير مدمج في Vercel حاليًا. للنشر الكامل ستحتاج:

- نشر FastAPI على خدمة منفصلة مثل Render/Fly/Railway/VPS.
- ضبط `VITE_API_BASE_URL` في الواجهة.
- تحديث CORS حسب domain الإنتاج.

```text
Taqreer AI
```

```text

```
